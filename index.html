iOS App Access Automation

Background

Apple restricts devices from installing and using apps from outside its official app market, App Store, without establishing a form of trust between the device and the app’s developer through the Apple Developer Program. Firebase simplifies this process.

However, when there’s a timezone difference, it can take up to 24 hours until the users can install the app.

This project aims to solve this problem by automating the work required on the developer’s side to grant access to the app after a user registers their device with Firebase.

Justification

There’s no question whether automating the process of building and distributing is worth it because doing it manually would cost a substantial amount of time to make small releases necessary for identifying issues early.

However, automating the registration of device UDIDs may not be necessary for every projects as it does not take no more than 10 minutes per device. Likewise, the email list required by the Firebase CLI for distribution can be stored as a simple text file in the git repository itself. So, adding a new email should also take no more than 10 minutes. Therefore, the total time investment per user would only be about 20 minutes. Even for 100 devices, the total cost would accumulate to just 33 hours (20 minutes × 100 devices / 60).

The reason this project automates these processes is that the KMP project was submitted as part of a university assignment. Since the university is in the UK and I live in Thailand, the timezone difference would cause delays for the reviewers. Furthermore, it is not known who will review my assignment or when. This makes it essential to automate all the work required to enable app access to the reviewers.

How It Works for Users

Once this project is functional, users will only need to wait approximately 1 hour after registering their device with Firebase, after which they will receive an email from Firebase that informs the build is ready to be downloaded and installed.

Personal Goal

Estimating something unknown has always been problematic for me. Sometimes, I estimate 16 hours and it's done in 4 hours. Sometimes, it's the other way around. This makes stakeholders unhappy because it's difficult for them to manage project timelines.

The project is being carried out with a focus on figuring out what can be done to address this weakness of mine and applying the findings along the way.

All lessons learned so far are highlighted under the section titled "Lesson Learned".

Design

The project consists of three main components:

  • iOSAppInviteAlert - uses Firebase Cloud Function to receive alerts when someone registers their device to Firebase.
  • iOSAppAccessAutomation - built with Node.js and Postgres, and deployed on Heroku. It manages the email list required by the Firebase CLI to distribute the build.
  • iOS Build GitHub Workflow - implemented using GitHub Actions to automate the creation of new provisioning profiles, building the app, and distributing it to testers.

Firebase (App Distribution) and Apple Developer Account (via Apple Connect API) are external services that do not require any coding.

Please note that the Node.js service in iOSAppAccessAutomation isn't necessary if the emails are managed as tester groups as the Firebase CLI allows distribution by groups. However, during the assignment, the automation aspect wasn't factored into implementation decisions, which led to the use of a Firebase invite link that doesn't add the user's email to a group. As a result, this automation project needs a service to manage an email list.

iOS Build GitHub Workflow is in iOSAppAccessAutomation repo. Security

Between iOSAppInviteAlert and iOSAppAccessAutomation

To ensure only iOSAppInviteAlert can save device information into iOSAppAccessAutomation, an asymmetric encryption called RSA is used.

iOSAppInviteAlert has a private key used for generating JWTs. The private key is saved in Firebase Cloud Secret Manager as iOSAppInviteAlert is a Firebase Cloud Function.

iOSAppAccessAutomation has a public key to verify the JWT sent from iOSAppInviteAlert. iOSAppAccessAutomation will reject a request to save device information if the JWT isn’t signed with the matching private key stored in Firebase Cloud Secret Manager. The public key is stored Heroku configuration variables as iOSAppAccessAutomation is deployed on Heroku.

For an organization, Heroku configuration variables may not be a viable option because Heroku doesn’t provide a way to pick and choose what features of Heroku a team member can have access to. This makes the secret keys visible to anyone who has access to Heroku dashboard. However, for this project, because I’m the sole engineer, this is not an issue.

As a final note on securing communications between iOSAppInviteAlert and iOSAppAccessAutomation, one improvement that can be done is to rotate the RSA keys to minimize the impact if they’re compromised. This will be done as part of learning secure infrastructure management though the exact date for this isn't set yet. Priorities are reviewed every 4 months.

Edge Case

If a user invites themselves using a new email but with a device that's already registered, Apple Connect API will return an error and iOSAppInviteAlert won't call the Github workflow to create a new build.

This is not a problem because the latest build will have already been built with a provisioning profile that's configured with the device's UDID since it's already registered.

However, the new email will replace the existing email in the Device table in iOSAppAccessAutomation so that the new email can receive notifications for new builds. It's designed this way because the history of device ownership is not important in this project.

As for the old email in User table, since it won't be cleared, it will continue to receive updates for new builds. This means the user can use either email to access the latest build.

Project Revision

Initially, it was designed to update the provisioning profile from the Firebase Cloud Function but later, it was found that Apple Connect API does not support updating provisioning profile. It only provides APIs for creating and deleting.

Because of it, it's decided to create a new provisioning profile for every build using Github Workflow instead.

Creating a provisioning profile returns the created provisioning profile as part of the response. Therefore, the task number 3 in the Task Breakdown was discarded.

The task number 7 is also canceled but because the time spent on this task revealed the missing information that led to this revision, it's still marked as 1 hour spent on this task.

This revision increased the project timeline by 13 hours.

Lessons Learned

  • The revision happened because during the design task, a third-party API was planned for use without knowing whether such API even exists. Just because APIs for downloading, creating and deleting exist, it cannot be assumed that an API for updating is available as well.
  • Estimation is easier when a big task is broken down into smaller ones. It makes it a lot easier to estimate and makes the estimations more accurate.
  • Dedicating time specifically for researching what needs to be done helps convert unknowns into knowns and improves the accuracy of estimates. Such research work should always be done before estimating anything in projects that involve unknowns.
